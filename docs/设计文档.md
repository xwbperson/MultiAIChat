# AI Workspace 设计文档

> **项目名称：** AI Workspace
> **定位：** 基于 Electron 的多站点 AI 聚合桌面客户端
> **最后更新：** 2026-07-11

---

## 一、项目概述

将 ChatGPT、Claude、DeepSeek 等多个 AI 网站聚合到一个桌面应用中，通过侧栏切换，提供统一、高效、隔离的多站操作体验。

### 核心目标

| 目标       | 说明                                            |
| ---------- | ----------------------------------------------- |
| 一站聚合   | 一个窗口访问 10+ 个 AI 网站，告别浏览器标签混乱 |
| 账户隔离   | 每个站点的登录态、Cookie、存储完全独立          |
| 多账户切换 | 同一站点可登录多个账号，一键切换不丢失状态      |
| 内存可控   | 支持 10+ 站点常驻而不撑爆内存                   |
| 代理灵活   | 逐站指定代理出口，适配多种网络环境              |

### 2.3 Session Partition 模型

这是实现账户隔离和多账户的核心机制：

```
Electron Session Partition
├── persist:chatgpt-free        ← ChatGPT 免费账号
├── persist:chatgpt-plus        ← ChatGPT Plus 账号
├── persist:chatgpt-team        ← ChatGPT Team 账号
├── persist:claude-personal     ← Claude 个人账号
├── persist:claude-work         ← Claude 工作账号
├── persist:deepseek            ← DeepSeek（单账号）
├── persist:kimi                ← Kimi（单账号）
└── persist:doubao              ← 豆包（单账号）
```

每个 partition 是一个完全隔离的 Chromium 会话，拥有独立的：
- Cookie
- localStorage / sessionStorage
- IndexedDB
- Cache
- Service Worker 数据

**切换账户 ≠ 重新登录**，而是切换到已有的 partition，所有状态完整保留。

### 3.2 标签页休眠机制

**背景：** 用户可能添加 10-20 个站点，每个 Chromium 页面占 200-400MB 内存，不加控制会消耗数 GB。

**设计：**

```
                    标签页生命周期状态机

  ┌────────┐   首次点击    ┌──────────┐
  │ 不存在  │───────────→│  创建加载  │
  └────────┘             └─────┬────┘
                               │
                               ▼
  ┌────────────────────  ┌──────────┐
  │                      │  活跃     │◄──── 点击切换回来
  │                      │ (内存常驻) │
  │                      └─────┬────┘
  │                            │
  │  切换到其他标签              │ 超过活跃上限
  │  且不在最近使用              │ 且一段时间未访问
  │                            ▼
  │  ┌──────────────────┐  ┌──────────┐
  │  │                  │  │  待休眠   │  30秒倒计时
  │  │                  │  │ (缓冲期)  │
  │  │                  │  └─────┬────┘
  │  │                    倒计时结束│
  │  │                            ▼
  │  │                    ┌──────────┐
  │  └───────────────────→│  休眠     │  保存URL，销毁WebContents
  │     点击唤醒           │ (内存释放) │
  │                       └──────────┘
```

**休眠调度策略：**

```
配置参数：
  MAX_ACTIVE   = 3    // 最多同时保持 3 个活跃页面
  IDLE_TIMEOUT = 30s  // 切走后 30 秒进入待休眠
  HIBERNATE_DELAY = 10s  // 待休眠后 10 秒确认休眠

调度规则：
  1. 当前页面 → 活跃（最高优先级）
  2. 最近访问的 2 个页面 → 保持活跃
  3. 其余页面 → 待休眠倒计时
  4. 倒计时到期 → 执行休眠（保存 URL，销毁 WebContents）
  5. 再次点击 → 用保存的 URL 重建 WebContents

休眠前后内存对比（15 个站点）：
  无休眠：15 × 300MB ≈ 4.5 GB
  有休眠： 3 × 300MB + 12 × 5MB ≈ 960 MB（节省 78%）
```

**休眠指示器：**

```
侧栏标签状态：
  ● 实心圆点     → 活跃页面
  ○ 空心圆点     → 已加载但非活跃
  ◌ 半透明圆点   → 休眠中（需重新加载）
  无圆点         → 未创建
  数字角标       → 有未读通知
```

### 3.3 多账户管理

**可行性评估：完全可行。** Electron 的 Session Partition 机制天然支持同一站点的多账户并行。

**数据模型：**

```
          "label": "Plus 账号",

        {
          "id": "chatgpt-team",
          "label": "Team 账号",
          "partition": "persist:chatgpt-team",

    },
    {
      "id": "claude",
      "name": "Claude",
      "url": "https://claude.ai",
      "color": "#d4a574",
      "icon": "🧠",
      "proxy": "http://127.0.0.1:7890",
      "accounts": [
        {
          "id": "claude-personal",
          "label": "个人",
          "partition": "persist:claude-personal",

          "partition": "persist:claude-work",

```

**账户切换机制：**

```
切换流程（无需重新登录）：

  当前：ChatGPT - 免费账号
        partition: persist:chatgpt-free
        
  用户点击：ChatGPT - Plus 账号
        ↓
  ① 隐藏当前 BrowserView（persist:chatgpt-free）
  ② 检查 persist:chatgpt-plus 是否已有 View
     ├── 有 → 直接显示
     └── 无 → 从休眠态恢复或新建
  ③ 更新侧栏高亮
  ④ 整个过程 < 500ms

  对比传统浏览器：
  ┌─────────────────────────────────────────────┐
  │ 浏览器方式              AI Workspace         │
  │                                             │
  │ 打开新无痕窗口     →    点击账户标签（<500ms） │
  │ 重新输入网址             自动恢复             │
  │ 重新登录                 已登录（cookie保留）  │
  │ 等待加载 5-10s           即时切换             │
  │ 无法同时使用             多账号并行           │
  │ 容易混淆窗口             清晰的标签区分        │
  └─────────────────────────────────────────────┘
```

**添加新账户流程：**

```
  侧栏长按站点图标 → 弹出菜单 → "添加账号"
        ↓
  新建 partition → 新建 BrowserView
        ↓
  自动打开目标站点登录页
        ↓
  用户正常登录
        ↓
  登录完成后，cookie 自动保存到该 partition
        ↓
  下次切换到该账号，无需重新登录
```

| `Ctrl+N`            | 窗口内 | 添加新站点                        |

## 六、代理架构设计

### 6.1 三层代理策略

```
用户视角的代理选择：

  方式一：TUN 模式（推荐）
  ┌──────────────────────────────────────────┐
  │  所有流量 → OS 虚拟网卡 → v2rayN/fclash   │
  │  AI Workspace 内不需要配置                 │
  │  优势：全局生效，零配置                     │
  └──────────────────────────────────────────┘

  方式二：逐站代理（精细控制）
  ┌──────────────────────────────────────────┐
  │  站点 A → http://127.0.0.1:7890 (fclash) │
  │  站点 B → socks5://127.0.0.1:10808       │
  │  站点 C → direct（直连）                   │
  │  每个站点独立配置                          │
  └──────────────────────────────────────────┘

  方式三：全局统一代理
  ┌──────────────────────────────────────────┐
  │  所有站点 → 同一个代理地址                  │
  │  设置一处，全部生效                        │
  └──────────────────────────────────────────┘
```

### 6.2 TUN 自动检测

```
应用启动时：

  检测系统 TUN 适配器
       │
       ├── 检测到 TUN → 弹窗提示"检测到 TUN 模式，建议站点设置为直连"
       │                 用户可一键将所有站点设为 direct
       │
       └── 未检测到 TUN → 检查是否有全局代理设置
                              │
                              ├── 有 → 按配置走
                              └── 无 → 各站点使用默认代理
```
